Blockers & Issues
BLOCKER: 7,848 unenriched grants (brisbane + qld-arts)
brisbane-grants (5,529) and qld-arts-data (2,319) have descriptions but 0% enrichment. These already have descriptions from APIs, so the enrichment pipeline skips them (it only targets grants with URLs but no descriptions). The batch enricher filters on enriched_at IS NULL AND url IS NOT NULL — but these grants may already have descriptions from the source API, so the enrichment path that scrapes URLs doesn't apply.

High Impact Low Effort
Fix: Run batchEnrich (Claude Haiku path) on these — it works on grants with descriptions but no enriched_at. Or simply mark them as enriched since they already have structured data from the source.
BLOCKER: 8,723 foundations stuck at "low" confidence
88% of foundations have profile_confidence: low. Only 1,244 (12.6%) have meaningful descriptions. The profiler runs at 5/batch in the pipeline — at that rate profiling 8,723 foundations would take ~1,745 pipeline runs (~36 days at 30-min intervals).

High Impact Medium Effort
Fix: Run a dedicated profiling batch with --limit=100 or higher. Parallelise across providers. The 4,953 foundations WITHOUT websites are the real bottleneck — they rely entirely on LLM knowledge + Gemini grounding.
ISSUE: GHL sync grants have no descriptions
121 grants from ghl_sync — only 1 has a description. These are CRM records (opportunity names) with no source URL. Neither enrichment path can help — no URL to scrape, no description for Haiku to analyse.

Low Impact Medium Effort
Fix: Use web search enrichment — take the grant name, web-search it, scrape the result. Or use Perplexity/Gemini grounded to look them up by name.
ISSUE: scrape-state-grants.mjs field name mismatch
The state scraper script (scrape-state-grants.mjs) maps plugin fields to title, source_url, source_id — but grant_opportunities uses name, url, source. The GrantEngine's discovery path normalises correctly, but this standalone script doesn't go through the engine.

Low Impact Low Effort
Fix: Fix column mapping in scrape-state-grants.mjs, or deprecate it since the pipeline-runner already runs discovery through the engine.
Improvement Opportunities
Priority 1: Batch-stamp brisbane/qld-arts as enriched
These 7,848 grants already have structured data from their source APIs (descriptions, categories, amounts). They don't need LLM enrichment. Just set enriched_at = now() on them — this will fix the 46% enrichment stat instantly to ~92%.

UPDATE grant_opportunities SET enriched_at = now() WHERE source IN ('brisbane-grants', 'qld-arts-data') AND enriched_at IS NULL;
Priority 2: Parallel foundation profiling
Current pipeline does 5 foundations per run. With 9 LLM providers, you could run 9 in parallel (one per provider) without hitting rate limits. The profiler already round-robins — just increase concurrency in build-foundation-profiles.mjs.

Target: profile 50-100 foundations per pipeline run → complete in ~88-175 runs (~2-4 days).
Priority 3: Enrich by web search for URL-less grants
121 GHL grants + some misc sources have no URLs. Add a third enrichment path: take the grant name → web search → scrape top result → extract with LLM. Could reuse the existing Perplexity or Gemini grounded providers since they have built-in search.

Pattern: enrichGrantBySearch(name) → Perplexity(name + "grant australia") → parse
Opportunity: Freshness tracking
No mechanism to detect stale/expired grants. The pipeline re-discovers but doesn't mark grants as closed when they disappear from source. Add a last_seen_at timestamp — if a grant isn't re-discovered in 2 runs, flag it as potentially expired.

This matters for user trust — showing expired grants damages credibility.
Opportunity: Incremental scraping (ETags/If-Modified)
Every pipeline run re-fetches all grants from all sources, even if nothing changed. State APIs (NSW ES, QLD CKAN) likely support If-Modified-Since or have updated_at sort fields. Use last_verified_at from previous runs to only fetch newer records.

Reduces API load, speeds up pipeline, and is more polite to government servers.
Opportunity: Dedup by content, not just URL
Current dedup is URL-based. But the same grant can appear from multiple sources with different URLs (e.g., a QLD grant on both qld.gov.au and business.gov.au). Embedding similarity could catch these — grants with >0.95 cosine similarity are likely duplicates.

You already have 100% embedding coverage — this is low-hanging fruit.
